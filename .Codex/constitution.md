# Project Constitution

> 이 문서는 Business OS 프로젝트의 불변 원칙을 정의한다.
> 모든 설계·구현·검증 전에 반드시 읽는다.
> 조항은 사용자 합의 없이 변경하지 않는다.

## 1. 프로젝트 원칙

### P-001. 고객 데이터 격리

모든 업무 데이터는 고객사(tenant) 경계를 넘어 노출되지 않아야 한다.

- 조회와 쓰기에는 tenant 범위를 명시한다.
- 고객 포털과 공유 링크는 추측하기 어려운 토큰과 서버 측 권한 검증을 사용한다.
- 로그, 알림, 파일 경로에서도 다른 tenant의 식별 정보가 섞이지 않게 한다.

**Gates 질문**: 이 변경이 tenant 간 데이터 노출 가능성을 만들지 않는가?

### P-002. 민감정보 보호

비밀번호, 토큰, API 키, 고객 개인정보를 소스·문서·Slack에 평문으로 기록하지 않는다.

- 시크릿은 환경별 비밀 저장소 또는 로컬 환경 파일로 관리한다.
- 예제와 로그에는 실제 고객 데이터를 사용하지 않는다.

**Gates 질문**: 저장소, 로그, 문서 또는 Slack에 민감정보가 남지 않는가?

### P-003. 모듈 재사용성

공통 기능과 업종별 기능을 분리하여 고객별 프로젝트에서 재사용할 수 있어야 한다.

- 인증, 사용자, 알림, 파일, 대시보드, 감사 로그는 공통 Core 후보로 관리한다.
- 업종별 차이는 공통 Core를 복제하지 않고 확장 모듈이나 설정으로 표현한다.

**Gates 질문**: 고객별 요구가 공통 Core의 불필요한 복제나 결합을 만들지 않는가?

### P-004. 검증 가능한 완료

기능은 완료 조건과 검증 결과가 있어야 완료로 간주한다.

- 모든 기능 작업은 수용 기준을 갖는다.
- 테스트 체계가 확정되기 전까지 최소한 재현 가능한 수동 검증 절차를 기록한다.
- 테스트 전략과 필수 자동화 범위는 `[TBD]`이며 첫 구현 spec에서 확정한다.

**Gates 질문**: 완료 조건과 그 조건을 확인한 증거가 있는가?

### P-005. 스펙 범위 준수

구현은 승인된 spec 범위를 벗어나지 않는다.

- 범위 밖 리팩터링, 기능 추가, 인프라 변경은 별도 작업으로 분리한다.
- 범위 변경이 필요하면 구현 전에 spec과 `.Codex/project.md`를 갱신한다.

**Gates 질문**: 이 변경이 합의된 목표와 완료 조건 안에 있는가?

### P-006. 외부 인터페이스 호환성

공개 API와 모듈 인터페이스의 호환성 수준은 `[TBD]`이며, 최초 공개 전 버전 정책을 확정한다.

- 확정 전에도 의도하지 않은 breaking change를 만들지 않는다.
- breaking change가 필요하면 영향 범위와 마이그레이션 방법을 plan에 기록한다.

**Gates 질문**: 인터페이스 변경의 영향과 전환 방법이 명시되었는가?

## 2. 예외 처리

1. plan의 `예외 사항`에 위반 조항과 근거를 적는다.
2. 위반 범위를 줄이는 대안을 함께 제시한다.
3. 사용자 확인 후 진행한다.
4. 구조적 충돌이면 spec 또는 constitution을 합의 후 개정한다.
