# Slack Workflow 운영 명세

> `#business_os`를 GitHub 이벤트, 의사결정, 블로커, 주간 리뷰의 운영 허브로 사용한다.
> 워크플로우는 아래 우선순위대로 하나씩 적용하고 완료 조건을 확인한 뒤 다음 단계로 이동한다.

## 공통 원칙

- **대상 채널**: `#business_os` (`C0BEFPB13QQ`)
- **기준 기록**: 코드·리뷰는 GitHub, 우선순위는 `.Codex/project.md`, 대화와 빠른 판단은 Slack에 둔다.
- **작업 식별자**: 모든 메시지에 `DEV-###`, `OPS-###`, `SPEC-###` 형식의 ID를 포함한다.
- **스레드 우선**: 자동 알림의 후속 논의는 원본 메시지 스레드에서 진행한다.
- **민감정보 금지**: 토큰, 환경 변수 값, 고객 개인정보, 원문 오류 로그를 폼과 메시지에 입력하지 않는다.
- **저소음 운영**: 사소한 커밋과 정상 로컬 로그는 전송하지 않는다.
- **확정 결정 회수**: Slack에서 확정된 결정은 Codex가 같은 작업 세션에서 GitHub 또는 `.Codex/` 문서에 반영한다.

| 순서 | 작업 ID | 워크플로우 | 우선순위 | 상태 | 다음 단계로 넘어가는 조건 |
|---|---|---|---|---|---|
| 1 | `OPS-002` | GitHub 알림 연동 | P0 | Ready | PR·CI·배포 테스트 알림 수신 |

| 4 | `OPS-005` | 주간 리뷰 자동 게시 | P2 | Backlog | 예약 게시와 프로젝트 문서 회수 검증 |
| 5 | `OPS-006` | 채널 탭 정리 | P2 | Backlog | 핵심 링크와 워크플로우 진입점 확인 |

## 1. GitHub 알림 연동 (`OPS-002`)

### 목적

PR, GitHub Actions 실패, 배포 상태를 채널에서 빠르게 인지한다.

### 설정

1. Slack 워크스페이스에 공식 GitHub 앱을 설치하고 비공개 저장소 접근을 승인한다.
2. `#business_os`에서 다음 명령을 실행한다.

```text
/github subscribe iamdeez/business-os pulls workflows deployments
```

3. `commits`, `issues` 알림은 초기 구독에서 제외한다.
4. 테스트 PR을 생성해 PR 생성·병합과 workflow 성공/실패 이벤트를 확인한다.

### 메시지 처리 규칙

- 실패한 workflow만 사람이 확인해야 하는 신호로 취급한다.
- PR 논의는 GitHub를 기준 기록으로 유지하고 Slack 스레드에는 결론과 링크만 남긴다.
- 배포 실패에는 관련 작업 ID, 영향, 다음 확인 시점을 스레드로 추가한다.

### 완료 조건

- [ ] 비공개 저장소 구독이 활성화된다.
- [ ] PR 생성 또는 병합 알림이 `#business_os`에 도착한다.
- [ ] workflow 테스트 이벤트가 도착한다.
- [ ] 메시지에 민감정보가 포함되지 않았음을 확인한다.

## 2. 결정·범위 변경 요청 (`OPS-003`)

### 목적

구현 도중 생긴 선택지와 범위 변경을 누락 없이 구조화하고 결정 이력을 회수한다.

### 트리거

Workflow Builder 링크 또는 채널의 **결정 요청** 바로가기에서 폼을 제출한다.

### 입력 필드

| 필드 | 필수 | 형식 |
|---|---|---|
| 작업 ID | 예 | `DEV-###`, `OPS-###`, `SPEC-###` |
| 결정 제목 | 예 | 한 문장 |
| 제안 | 예 | 권장 선택안 |
| 이유 | 예 | 목표·근거·제약 |
| 대안 | 아니요 | 검토한 다른 선택지 |
| 영향 | 예 | 범위·일정·비용·데이터·호환성 |
| 결정 기한 | 예 | 날짜와 시간 |
| 관련 링크 | 아니요 | PR, spec, 오류 재현 문서 |

### 출력 형식

```text
🟡 결정 요청 · {작업 ID} · {결정 제목}
제안: {제안}
영향: {영향}
결정 기한: {기한}
관련 링크: {링크}
```

### 상태 규칙

- 최초 메시지는 `🟡 결정 요청`으로 게시한다.
- 논의는 메시지 스레드에서 진행한다.
- 결정 후 스레드에 `🟢 Accepted` 또는 `⚪ Deferred`와 근거를 남긴다.
- Accepted 결정은 `.Codex/project.md` 결정 로그 또는 `.Codex/templates/decision.md` 기반 문서로 회수한다.

### 완료 조건

- [ ] 필수 필드가 비어 있으면 제출할 수 없다.
- [ ] 테스트 제출이 지정 채널에 게시된다.
- [ ] 스레드에서 결정 상태를 남길 수 있다.
- [ ] 확정 결정을 로컬 문서로 회수하는 절차가 검증된다.

## 3. 블로커 신고 (`OPS-004`)

### 목적

진행을 멈춘 문제를 일반 진행 보고와 분리하고 해소에 필요한 행동을 명확히 한다.

### 트리거

Workflow Builder 링크 또는 채널의 **블로커 신고** 바로가기에서 폼을 제출한다.

### 입력 필드

| 필드 | 필수 | 형식 |
|---|---|---|
| 심각도 | 예 | P0 서비스 중단 / P1 작업 중단 / P2 우회 가능 |
| 작업 ID | 예 | 프로젝트 작업 ID |
| 현상 | 예 | 민감정보를 제거한 요약 |
| 영향 | 예 | 멈춘 작업과 사용자 영향 |
| 확인한 내용 | 예 | 재현·진단 요약 |
| 필요한 도움 | 예 | 결정, 권한, 계정, 기술 지원 중 하나 이상 |
| 관련 링크 | 아니요 | PR, Actions, 문서 링크 |
| 다음 확인 시점 | 예 | 날짜와 시간 |

### 출력 형식

```text
🔴 블로커 · {심각도} · {작업 ID}
현상: {현상}
영향: {영향}
필요한 도움: {필요한 도움}
다음 확인: {시점}
```

### 상태 규칙

- 신고 즉시 `.Codex/project.md`의 해당 작업을 Blocked로 옮긴다.
- 진행 업데이트는 원본 메시지 스레드에 남긴다.
- 해소되면 `🟢 해소 · {해결 요약} · {후속 작업}`을 남기고 보드를 갱신한다.
- 실제 토큰, 고객 데이터, 전체 로그는 붙이지 않는다.

### 완료 조건

- [ ] 심각도와 필요한 도움이 필수 입력으로 설정된다.
- [ ] 테스트 신고가 지정 채널에 게시된다.
- [ ] Blocked → 해소 → 보드 복귀 절차가 검증된다.

## 4. 주간 리뷰 자동 게시 (`OPS-005`)

### 목적

한 주의 결과와 리스크를 정리하고 다음 주 우선순위를 세 개로 제한한다.

### 스케줄

- 매주 금요일 17:00, `Asia/Seoul`
- 공휴일이나 휴무일에는 직전 업무일에 수동 실행한다.

### 자동 게시 내용

```text
📅 Business OS 주간 리뷰
1. 이번 주 Done과 검증 결과는 무엇인가?
2. Blocked와 해소 조건은 무엇인가?
3. 새로 확정된 결정 또는 범위 변경은 무엇인가?
4. 다음 주 최우선 결과 세 개는 무엇인가?
```

### 회수 절차

1. 답변은 자동 게시 메시지의 스레드에 작성한다.
2. Codex가 `.Codex/templates/weekly-review.md` 형식으로 정리한다.
3. `.Codex/project.md`의 보드, 리스크, 현재 초점을 갱신한다.
4. 문서 변경은 브랜치·PR 규칙에 따라 병합한다.

> Slack Workflow Builder만으로 로컬 저장소를 직접 갱신하지 않는다. 문서 반영은 Codex 작업으로 수행한다.

### 완료 조건

- [ ] 지정 시간에 테스트 예약 메시지가 게시된다.
- [ ] 스레드 답변을 주간 리뷰 문서로 변환한다.
- [ ] 프로젝트 보드와 리스크가 같은 내용으로 갱신된다.

## 5. 채널 탭 정리 (`OPS-006`)

### 목적

새 대화나 검색 없이 프로젝트 기준 기록과 워크플로우에 접근한다.

### 권장 탭 순서

1. **메시지**: 운영 대화와 자동 알림
2. **프로젝트 허브**: `.Codex/project.md` 또는 GitHub 렌더링 링크
3. **SPEC-001**: 현재 MVP spec 링크
4. **GitHub**: 저장소와 열린 PR 링크
5. **워크플로우**: 결정 요청, 블로커 신고, 주간 리뷰 바로가기

### 운영 규칙

- 탭에는 공개 가능한 링크만 사용한다.
- 현재 spec이나 프로젝트 허브가 바뀌면 같은 작업에서 탭도 갱신한다.
- Canvas를 사용할 수 없는 요금제라면 채널 북마크 또는 고정 메시지로 동일한 순서를 유지한다.

### 완료 조건

- [ ] 다섯 개 진입점이 채널 상단에서 접근 가능하다.
- [ ] 권한 없는 사용자나 외부 사용자에게 비공개 링크가 노출되지 않는다.
- [ ] 워크플로우 바로가기가 올바른 채널로 게시한다.

## 운영 점검표

각 워크플로우 적용 후 아래 항목을 확인한다.

- [ ] 예상한 트리거에서 한 번만 실행된다.
- [ ] 출력이 `#business_os` 또는 지정 스레드로 전달된다.
- [ ] 작업 ID와 관련 링크로 원본 기록을 추적할 수 있다.
- [ ] tenant 정보, 토큰, 고객 개인정보가 포함되지 않는다.
- [ ] GitHub·`.Codex/project.md`와 상태가 충돌하지 않는다.
- [ ] 실패 시 수동 대체 절차가 있다.
